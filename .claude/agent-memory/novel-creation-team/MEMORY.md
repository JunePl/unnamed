## Lessons Learned
- Large docs (1000+ lines): write in sections via Bash heredoc
- Read files in sections (offset/limit) when >25000 tokens
- Always verify line counts with wc -l before/after edits
- Files >280KB: use Bash tail/grep instead of Read tool
- Mian.novel is ABSOLUTE BASELINE for Ch1-4 content
- When splitting chapters, verify total count constraint first
- Absorbing content into existing chapters avoids numbering cascades
- **★ 彼界不应有现实线索 ★**: 用户核心反馈——"彼界为什么会有现实的线索？"→超长线应完全独立
- **★ 概念必须make sense ★**: "开采"概念被否决因为用户问"彼界为什么会被开采？"→所有概念需要内在逻辑自洽
- **★ 新线路要有独特张力类型 ★**: Line E(赛琳)=世界张力，不与A(悬疑)/B(政治)/D(命运)重叠
- **★ 总纲写入方法论(2026-03-02) ★**: 分段写入(每段250-350行)效率最高。强制agent"不要问问题直接写入"否则会无限循环提问。V4.0用5个agent段完成2046行
- **★ 后台agent编辑权限问题(2026-03-02) ★**: novel-creation-team后台agent对总纲文件Edit权限被拒绝。解决方案：直接在主agent中执行Edit操作，不使用后台agent做文件修改
- **★ 场景节拍零心理描写(2026-03-05) ★**: 用户明确要求"不需使用心理描写，角色可以有心理但是我们不能写出来"。场景节拍只用客观动作/事件/对话/plot progression
- **★ 灵质体≠灵晶(2026-03-05) ★**: 这两个概念被混用了15+处。灵质体=死亡残留。灵晶=灵质凝结。三密钥灵晶=特殊。阿尔温只有一颗灵晶，整颗交给奥古斯都。不存在"分散保管"
- **★ 案件集→第一卷同步冲突(2026-03-05) ★**: 案件集V5.1设定Ch27=王都，但第一卷V5.9 Ch27=提耶林。解决：用户选择以案件集为准重写。教训=上下游文件同步必须及时，否则积累大量重写工作
- **★ 灵名≠灵契类型(2026-03-10) ★**: 蚀析=灵的名字(白色蜘蛛)，噬暗源流=灵契源流类型。用户说"漂亮的白色蜘蛛"时确认蚀析是灵的外观/名字。曾有30+处错误将"噬暗·蚀析型"当作能力类型名
- **★ 大文件分段写作效率优化(2026-03-10) ★**: 4111行文件=11个分段(150-550行/段)+1个保留段。先写各段到/tmp→术语验证→concatenate→copy。比直接编辑大文件效率高10倍
- **★ 角色身份不能自行发明(2026-03-10) ★**: 苏林被我写成"护卫"但用户说"你自己写的"。教训：用户没有明确设定角色身份时，不要自行发明专业身份。应该问用户或保持模糊
- **★ 彼界逻辑严格性(2026-03-10) ★**: 用户问"可是所有人都是随机出现在彼界"直接否决了苏暮雪"来研究彼界遗迹"的设定。教训：彼界的核心规则（随机入口+灵魂体）不能被角色motivation便利性绕过
- **★ Python \b与中文字符(2026-03-12) ★**: Python 3中`\b`不会在数字和中文字符之间触发(两者都是`\w`)。`Ch53的`中53和的之间无word boundary。解决方案：用`Ch(\d+)(?!\d)`替代`Ch(\d+)\b`，或使用定向字符串替换
- **★ 章节重编号double-shift防护(2026-03-12) ★**: 通用正则映射脚本会将已更新的值再次映射(如Ch53→Ch55→Ch57)。解决方案：①一次性运行·不重复 ②使用定向字符串替换(28个精确目标)替代通用映射 ③从底部向顶部修改避免行号漂移
- **★ 后台agent写入支线登记表失败(2026-03-12) ★**: background agent对支线登记表的编辑未完成。解决方案：关键文件修改在主agent中直接执行。后台agent仅适用于独立性强、失败可容忍的任务
