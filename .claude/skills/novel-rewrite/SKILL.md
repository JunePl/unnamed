# 正文写作 / 重写工作流程（98% 贴合度铁律 + 三段法）

**触发条件：** 用户要求修改、重写、新写、续写 `/root/Pi/unnamed/正文/` 目录下的任何小说正文。

**核心铁律：** 每一句话和每一个单词之间的组合都必须能在参考材料、调研报告、以及锚定的两本书（一世之尊 + 术师手册，或场景对应的其他参考）里找到对应组合。**不能 98% 贴合就不写。**

**2026 神殿章血泪经验**：单纯的"写完再 grep 自检"不够。**写之前的"节奏分析+骨架搭建"才能从源头避免 AI 默认偏好**。

---

## ⛔ 三条不可绕过的硬验证

### 硬验证 1：每次提交样本必带"已读参考清单"

```
【已读参考】
- 文件 1：[路径] —— 一句话总结这次读到什么有用的句式/语感
- 用户已发布正文 grep 结果：[搜了什么关键词] —— [找到了哪些可借鉴的句子，给出行号]
```

没有这段清单 = 没读 = 不能交稿。

### 硬验证 2：每次提交样本必带"grep 自检日志原始输出"

不是"我跑了，全 0"声明——是终端实际输出。

### 硬验证 3：用户负反馈"差的很远"出现时——禁止局部修补

第一次：5 分钟内重新检查视角/节奏/拍点三件事并向用户汇报。

第二次：禁止继续写样本。停下来让用户口头解释他想要的是什么，**用文字复述给他确认**，确认后再写。

## 流程总览（按顺序，不可跳步）

```
Step 0: 问问题（确认 95% 明白）
Step 1: 读所有相关参考 + grep 用户已发布正文找同类描写
Step 2: 列叙事拍点骨架（每拍"想要 X 但 Y 所以 Z"）
Step 2.5: 节奏分析（场景类型 + 节奏定调 + 威胁级别 + 参考锚点）⭐ 2026 新增
Step 3: 句式骨架搭建（每句话先列骨架，grep 验证 100% 贴合参考）⭐ 2026 重写
Step 4: 填词写作（按骨架填词，每个不寻常的词 grep 验证，每 ≤150 字给用户看）⭐ 2026 重写
Step 5: 调用 novel-rewrite-verify skill 跑独立验证 ⭐ 2026 新增
Step 6: 根据验证结果修复
Step 7: 用户确认后用 Edit 工具写入
```

## Step 0 — 接到任务先问问题（不可跳过）

按 CLAUDE.md 项目级铁律——确保 95% 以上明白用户意思。

**必问的问题类型：**
1. 剧情机制（关键动作的物理/逻辑机制）
2. 角色心理动机（情绪锚点）
3. 影响范围（哪段重写、哪段保留、后续是否需要联动修改）
4. AI 浓度判定（用户说"AI 浓度高"指的是哪一类问题）
5. 角色能力/职业/设定

**问题数量 3-6 个。**

## Step 2 — 列每一拍的"想要X但Y所以Z"骨架（叙事骨架，不是句式骨架）

| 拍 # | 角色想要做什么 | 阻碍是什么 | 反应是什么 | 视觉/恐怖细节 | 情绪化动词 |
|---|---|---|---|---|---|

**每一拍至少满足：1 个情绪化动词 + 1 个具体视觉细节。**

## Step 2.5 — 节奏分析（2026 新增，不可跳过）

**Why：** 之前我"先想内容再凭感觉填句子"——句式骨架是凭感觉写的，所以经常出短独段堆砌/破折号过多/cliché 比喻。**这一步强制把"节奏"作为独立决策**。

### 蓝图必含 4 项

1. **场景类型**：日常对话 / 战斗紧张 / 感官冲击 / 内心独白 / 探索观察 / 高密度信息 / 黑色幽默 / 余韵转场
2. **节奏定调**（参考 [[feedback_skeleton_first]] §B）：
   - 中长句连缀（默认基础——听日/竹宫风格）
   - 戏剧短独段间插（少量，仅 1-2 处揭示节点）
   - 对话密集
   - 挣扎结构（每拍"想要 X 但 Y 所以 Z"）
3. **威胁级别**（如果场景含危险）：L0-L4（参考 [[feedback_scene_consistency]] §B），决定身体反应强度
4. **参考锚点**：明确这段对应**哪个参考作品的哪个场景**——后续骨架/填词都从这个参考找

**输出格式**：500 字以内的节奏蓝图，给用户确认。

### 节奏蓝图示例

> **场景类型**：探索观察 + 致命压感击退
> **节奏定调**：中长句连缀（默认）+ 2-3 处戏剧短独段（揭示节点：发现腿/识别压感=死）
> **威胁级别**：L3（致命压力——身体冻结、不敢动、呼吸停）
> **参考锚点**：
>  - 探索部分 → 默读"鞋很干净"式工作流观察
>  - 致命压感 → 术师手册"喘不过气"+ 用户正文 2994"肺里的气一下子被碾空"
>  - 不能用 → 术师手册的"无形的手攥住" cliché

## Step 3 — 句式骨架搭建（2026 重写，核心创新）

**核心铁律：每一句话的句式骨架必须能在参考里 grep 到 ≥1 完全相同结构的句子。**

### 操作流程

1. **按 Step 2.5 节奏蓝图，列每一句的骨架结构**：
   - "X 在 Y 的时候，Z 也 W"
   - "X 想 Y，但 Z"
   - "X 没 Y，先 Z"
   - "X 的 Y 一下子被 Z"

2. **对每一个骨架 grep 参考**：
   ```bash
   REF="/root/Pi/unnamed/Project/Reference/术师手册_utf8.txt \
        /root/Pi/unnamed/Project/Reference/一世之尊_utf8.txt \
        /root/Pi/unnamed/Project/Reference/文风参考.md \
        /root/Pi/unnamed/正文/正文.novel"
   grep -h "X.{0,3}Y.{0,5}时候.{0,3}Z" $REF | head -5
   ```

3. **0 命中 = 骨架自造，必须替换**——找参考里真实存在的相似骨架替换

4. **输出格式**：

| 拍# | 我想写的内容 | 句式骨架 | grep 命中（参考行） |
|---|---|---|---|
| 1 | 阿尔温背靠神像 | "X 这才发觉 Y" | 术师手册 L###："...这才发觉..." |
| ... | ... | ... | ... |

详见 [[feedback_skeleton_first]] §A.2 + §C。

## Step 4 — 填词写作（2026 重写）

**核心铁律：每个不寻常的动词/形容词/名词都必须 grep 验证。**

### 操作流程

1. **按 Step 3 确认的骨架填词**
2. **每写一段 ≤150 字立刻给用户看**——不写完整段，控制错误密度
3. **每个不寻常词**（不是"的/是/在/他/她"等功能词）都 grep：
   ```bash
   grep -h "<词>" $REF | head -3
   ```
4. **0 命中 = 自造词，必须替换**——找参考里真实存在的同义词
5. **特别注意前文已用术语**：grep 用户已发布正文，不能改说法（如用户写"灵纹"我不能写"蛛印"）

详见 [[feedback_skeleton_first]] §A.3 + §D。

---

## Step 5 — 调用 novel-rewrite-verify skill（2026 新增）

**Why：** 由独立 skill 跑验证，避免"写作的我"做"验证的我"——分工避免认知盲点。

写完一段（或整段）后：

```
调用 Skill: novel-rewrite-verify
输入: 刚写的段落（写到 /tmp/check.txt）
处理: 自动跑所有 feedback 检查
输出: 违规清单 + 修改建议
```

`novel-rewrite-verify` 会跑：
- 所有黑名单 grep（包括 2026 新增的 cliché 比喻）
- 短句独段堆砌检测
- "立刻命名"/"按理说"等凭空植入检测
- 反应强度匹配检测
- 物理逻辑检测
- 伏笔明示检测

---

## Step 6 — 根据验证结果修复

根据 verify 输出的违规清单**逐条**修复。修复后**再次调用 verify** 直到 0 违规。

---

## Step 7 — 用户确认后写入

只有用户说"开始"/"写入"/"OK 写"之后才能动 Edit/Write。

**写入流程：**
1. 用 Edit 工具替换目标段
2. 如果原文带【要改：…】标记，整块替换包括标记本身
3. 联动修改必须一并完成
4. 写入后用 Read 工具或 grep 确认结果

**写入后汇报必须包括：**
- 改动了哪些行
- 自检最终汇总
- 还有哪些后续可能要改但不在本次范围内的
